Chapter 1 · Part I — Foundations

    
Client and server, plainly explained

    
Two programs, one conversation, on top of HTTP.

  


  

    A web application is not a single program. It is two programs that talk to
    each other. One runs in the user's browser — we call it the
    client. The other runs on a machine somewhere on the
    network — we call it the server. They are
    written in different folders, deployed separately, and often by different
    people. They communicate over a protocol called HTTP: a
    text-based question-and-answer format. The client asks; the server answers.
  


  

   Browser (web/)                                Server (api/)
   ┌──────────────────┐    HTTP request          ┌──────────────────┐
   │                  │ ───────────────────────▶ │                  │
   │  React app       │                          │  Express app     │
   │  TypeScript      │    HTTP response         │  TypeScript      │
   │                  │ ◀─────────────────────── │                  │
   └──────────────────┘                          └────────┬─────────┘
                                                          │
                                                          ▼
                                                 ┌──────────────────┐
                                                 │  PostgreSQL DB   │
                                                 └──────────────────┘
  


  
Why split them at all?

  

    Because they have different jobs and different threats. The client renders
    pixels and reacts to clicks — it lives on a device you do not control,
    and anything it knows, the user knows. The server is the trustworthy half:
    it holds the database password, hashes user passwords, signs tokens, and
    decides who may see what. The split is not a convenience; it is a
    security boundary.
  


  

    Why
    Anything sensitive (DB credentials, JWT signing secret, bcrypt rounds) lives
    only in the server. The browser never even sees these strings. If you ever
    find yourself wanting to put a password into client code, stop — you
    are about to ship that password to every user.
  


  
What flows between them?

  

    Just text. The client sends an HTTP request — a method (GET,
    POST, PATCH, DELETE), a path
    (/api/v1/users), a few headers, and an optional JSON body. The
    server sends back a status code (200, 401,
    500) and a JSON response. That is it. There is no magic
    pipe.
  


  

    Here is a real request from this project's LoginForm:
  



POST /api/v1/auth/login HTTP/1.1
Host: localhost:3000
Content-Type: application/json
Content-Length: 64

{"email":"jane@example.com","password":"hunter2hunter"}


  
And the matching response:



HTTP/1.1 200 OK
Content-Type: application/json

{
  "accessToken": "eyJhbGc...",
  "refreshToken": "kT4o9...",
  "tokenType": "Bearer",
  "expiresIn": 900,
  "user": { "id": "...", "email": "jane@example.com", "roles": ["STUDENT"] }
}


  

    Every interesting thing in this handbook — logging in, listing users,
    editing a profile — reduces to a request like the one above and a
    response like the one below. The complexity is not in the wire format; it
    is in deciding what to send, who may send it, and what to
    do when something goes wrong.

Chapter 2 · Part I — Foundations

    
Why TypeScript? Types as a second pair of eyes

    
A compiler that reads your code before your user does.

  


  

    JavaScript will happily let you write user.emial. The bug
    surfaces only when a real user logs in, fails, and complains. TypeScript
    is JavaScript plus a type-checker — a program that
    reads your source and refuses to compile if you mis-spell a field, pass
    the wrong shape, or forget to handle null.
  


  

    It is not a different language. Strip the type annotations from any
    .ts file in this project and you have ordinary JavaScript. The
    types exist only at build time; at runtime they are gone.
  


  
The compact version of the syntax


// Variables
const port: number = 3000;
const name: string = 'Jane';

// Function parameters and return type
const greet = (who: string): string => `Hello, ${who}`;

// Object shapes
interface User {
  id: string;
  email: string;
  isActive: boolean;
}

// Nullability is explicit — "maybe absent" must be spelled out
let lastLogin: Date | null = null;

// Unions: a value that can be one of several things
type Role = 'ADMIN' | 'TEACHER' | 'STUDENT';

// Generics: a type that depends on another type
type ApiResult<T> = { ok: true; data: T } | { ok: false; error: string };


  
What types catch that runtime tests don't

  

    Consider this real snippet from the server's user service. Notice the
    return type: PublicUser | null.
  


  
api/src/modules/auth/service.ts


getUserPublic: async (userId: string): Promise<PublicUser | null> => {
  const user = await authRepository.findUserById(userId);
  if (user === null || user.deletedAt !== null) {
    return null;
  }
  return toPublicUser(user);
},


  

    Any caller that uses the return value must handle the
    null case — the compiler will not let them dereference
    .email until they have proved the value is not
    null. That is a whole class of TypeError: Cannot read
    properties of null errors that simply cannot ship.
  


  

    Why
    The compiler is a tireless reviewer. Every rename, every refactor, every
    "I added a field to the database" gets a wave of red squiggles in your
    editor until you have updated every place that touches the changed shape.
    A team without types finds those places by deploying and waiting for
    bug reports.
  


  
Types do not protect you from outside data

  

    The compiler can prove things about code you wrote; it cannot prove
    anything about JSON arriving from the network. When the browser sends
    { "email": 42 }, the TypeScript type says
    email: string — but the value is a number, and your
    server is now in trouble.
  

  

    That is why we use Zod at every boundary: it parses
    untrusted input at runtime and produces a value the compiler can
    then trust. We will meet Zod in Chapter 9.
  


  

   ┌─────────────────┐    raw bytes     ┌─────────────────┐
   │  Network /      │ ───────────────▶ │  Zod parser     │  ← runtime check
   │  database /     │                  │  (validate.ts)  │
   │  env vars       │ ◀─────────────── │                 │
   └─────────────────┘    typed value   └────────┬────────┘
                                                 │
                                                 ▼
                                        ┌─────────────────┐
                                        │  Your code      │  ← compile-time
                                        │  (TS types)     │     check
                                        └─────────────────┘
  


  
The mental model

  

    Think of TypeScript types as contracts. A function signature is a
    promise: "give me these inputs and I will give you back this output,
    or I will throw." The body of the function is then checked against
    that promise. Other code that calls it relies on the promise without
    having to read the body.
  


  

    Pitfall
    Resist the urge to write any. It is an escape hatch that
    silences the compiler — useful occasionally, lethal as a habit. Each
    any punches a hole in your type system through which bugs
    will eventually crawl. Prefer unknown when you genuinely don't
    know the shape; the compiler will then force you to narrow it before use.

The vocabulary that holds it together

  

    
Module

    
A self-contained business domain. auth/ handles login and tokens; users/ handles CRUD on user records. A module owns its own service, repository, routes, and DTOs.

    
Controller

    
The thin layer that touches req and res. It unpacks the request, calls the service, and sends the response. Almost no logic.

    
Service

    
The brain. Business rules live here: "admins can delete anyone except themselves", "a reused refresh token revokes the entire session". The service is unit-testable without an HTTP server.

    
Repository

    
The data access layer — the only place that imports the Prisma client. If you ever swap PostgreSQL for something else, this is the single layer that changes.

    are hashed? Only the service (and a util) cares. Want to add an index?
    Only the repository. Layers are not bureaucracy — they are
    seams, places the code is designed to be pried apart at.

Chapter 4 · Part II — The Server

    
Process startup & graceful shutdown

    
How a Node program becomes a long-lived web service.

  


  

    A Node.js HTTP server is, at its core, a single process that calls
    server.listen(port) and then waits. When you ran
    node script.js in your first programming class, the program
    ended once the last line of code finished. A server program is different:
    it sits in an event loop, taking requests, until something tells it to
    stop. Knowing how it starts — and how it stops cleanly — is
    the first thing to understand about api/.
  


  

    Open api/src/index.ts. This is the file Node runs when you
    execute pnpm dev.
  


  
api/src/index.ts


const start = (): Server => {
  const app = createApp();
  const server = app.listen(env.PORT, () => {
    logger.info({ port: env.PORT, env: env.NODE_ENV }, 'server started');
  });
  return server;
};


  

    Three things happen here. createApp() builds the configured
    Express application (Chapter 5). app.listen() binds a
    TCP socket to the port from environment configuration. The callback fires
    once the socket is open — we log that we're alive, then return the
    server handle so the rest of the file can hold on to it.
  


  
The shutdown problem

  

    What happens when the operating system tells the process to stop? Most
    student projects either ignore the question or do this:
  



process.on('SIGTERM', () => process.exit(0));


  

    That kills in-flight requests mid-response, leaves database connections
    dangling, and shows partial pages to the user who clicked just before the
    deploy. We can do better. Here is how the project does it:
  


  
api/src/index.ts


const shutdown = async (server: Server, signal: string): Promise<void> => {
  logger.info({ signal }, 'shutdown initiated');

  // Stop accepting new connections; let in-flight requests finish.
  const closeServer = new Promise<void>((resolve, reject) => {
    server.close((err) => (err ? reject(err) : resolve()));
  });

  // Hard cap — kills hung connections after the deadline.
  const timeout = new Promise<never>((_, reject) => {
    setTimeout(() => reject(new Error('shutdown timed out')), 10_000).unref();
  });

  try {
    await Promise.race([closeServer, timeout]);
    await disconnectPrisma();
    process.exit(0);
  } catch (err) {
    logger.error({ err }, 'shutdown error');
    process.exit(1);
  }
};


  

    Read it as a small story. We stop accepting new requests by
    calling server.close(); existing requests get to finish. We
    race that against a ten-second timeout because some requests may hang
    forever (a database query that never returns). Whichever finishes first
    wins. Either way, we then close the database connection pool and exit.
  


  

    Why a timeout?
    A graceful shutdown that waits forever is just a hang. The orchestration
    layer (Docker, Kubernetes, systemd) will eventually SIGKILL us
    anyway — this way we control the deadline and log a clear error,
    instead of vanishing without explanation.
  


  
The signals we listen for


for (const signal of ['SIGTERM', 'SIGINT'] as const) {
  process.on(signal, () => { void shutdown(server, signal); });
}

process.on('uncaughtException',  (err) => { logger.fatal({err}); process.exit(1); });
process.on('unhandledRejection', (err) => { logger.fatal({err}); process.exit(1); });


  

    
SIGTERM
"Please stop, but cleanly." Sent by Docker, Kubernetes, systemd on a normal restart.

    
SIGINT
What you generate when you press Ctrl-C in your terminal.

    
uncaughtException
A synchronous error nobody caught. The process is in an unknown state; the only safe move is to exit and let the orchestrator restart us.

    
unhandledRejection
An async error nobody caught. Same logic.

  


  

    Pitfall
    Some tutorials catch uncaughtException and keep running.
    Don't. By the time you reach that handler, your application's invariants
    are broken — you can no longer trust any variable. Log loudly, exit
    fast, and let the supervisor restart you with a fresh process.

Chapter 5 · Part II — The Server

    
Composing the Express application

    
Middleware is a pipeline. Order is everything.

  


  

    Express is a thin library around Node's HTTP module. Its single big idea
    is middleware: a function that receives the request, can
    mutate it, and then either responds or calls next() to pass
    control to the next function in the chain. An Express application is
    a list of middleware ending in a route handler.
  


  

    Open api/src/app.ts. The whole file is one function,
    createApp(), that builds and returns an application. Putting
    it in a function (rather than at module scope) makes the app
    composable — tests can build a fresh app per test, with
    different env or different routes.
  


  
api/src/app.ts


export const createApp = (): Application => {
  const app = express();

  app.disable('x-powered-by');
  app.set('trust proxy', 1);

  app.use(helmet());
  app.use(cors({ origin: env.CORS_ORIGINS, credentials: true }));
  app.use(express.json({ limit: env.JSON_BODY_LIMIT }));
  app.use(express.urlencoded({ extended: false, limit: env.JSON_BODY_LIMIT }));
  app.use(compression());

  app.use(requestId);
  app.use(httpLogger);

  app.use(rateLimit({ windowMs: env.RATE_LIMIT_WINDOW_MS, limit: env.RATE_LIMIT_MAX, ... }));

  app.get('/health',       (_req, res) => { res.json({ status: 'ok' }); });
  app.get('/health/ready', async (_req, res) => { /* DB check */ });

  app.use('/api/v1/auth',  authRouter);
  app.use('/api/v1/users', usersRouter);

  app.use(notFoundHandler);
  app.use(errorHandler);

  return app;
};


  

    Read this top to bottom. The order is not arbitrary; each call adds a
    layer to a pipeline that every request travels through.
  


  
The pipeline, step by step


  
    
    
  1. 
      app.disable('x-powered-by') — Express
      adds an X-Powered-By: Express header by default. That tells
      every drive-by scanner exactly what they are attacking. Off.
    
    2. 
    
  2. 
      app.set('trust proxy', 1) — tells
      Express that one layer of reverse proxy (nginx, Cloudflare) sits in
      front. Without this, req.ip would be the proxy's IP, not the
      real client.
    
    3. 
    
  3. 
      helmet() — adds a bundle of security
      headers (X-Content-Type-Options, Strict-Transport-Security,
      and others). Many small wins.
    
    4. 
    
  4. 
      cors(...) — allows the browser at
      http://localhost:5173 (the Vite dev server) to call this
      API. The origin list is an explicit allowlist from env; we
      never use * with credentials.
    
    5. 
    
  5. 
      express.json({ limit }) — parses
      JSON request bodies, capped at JSON_BODY_LIMIT bytes (default
      100KB). The limit blocks a denial-of-service vector: a megabyte
      POST can otherwise tie up memory while it parses.
    
    6. 
    
  6. 
      compression() — gzips responses
      large enough to be worth it. Pure win for the network.
    
    7. 
    
  7. 
      requestId — attaches a unique ID to
      every request. Logs, errors, and the response header all carry it, so
      one user complaint becomes one greppable string. (Chapter 13.)
    
    8. 
    
  8. 
      httpLogger — structured access log
      via pino. After the request ID so the ID is included.
    
    9. 
    
  9. 
      rateLimit(...) — a global throttle:
      100 requests per minute per IP. Login and register have stricter
      limits applied locally (Chapter 10).
    
    10. 
    
  10. 
      Health endpoints — /health answers
      "is the process up?" and /health/ready answers "can it
      serve traffic?" (by pinging the DB). Kubernetes uses both.
    
    11. 
    
  11. 
      Module routers — the real endpoints. Each module
      exports a Router that gets mounted at a path prefix.
    
    12. 
    
  12. 
      notFoundHandler — if no route
      matched, return a 404 in the same JSON shape as every other error.
    
    13. 
    
  13. 
      errorHandler — catches everything
      thrown anywhere upstream and produces a consistent error response.
      Must be last. Chapter 12.
    
    14. 
  


  
Why order matters


  

    Pitfall
    Put express.json() after a route and that route will receive
    an empty req.body. Put requestId after
    httpLogger and the log entries won't have an ID. Put the
    error handler before the routes and it will never see their errors.
    Middleware order is part of the program's semantics, not its
    style.

Configuration & environment validation

    
Fail at startup, not in production.

  


  

    Every non-trivial server reads configuration from process.env:
    the database URL, JWT secrets, the port to bind, log level, allowed CORS
    origins. The naive approach — sprinkle process.env.WHATEVER
    across the codebase — has three problems:
  


  
    
    
  1. Every read returns string | undefined. The compiler has no idea what's defined.
    2. 
    
  2. A typo in the variable name fails silently — you read undefined and carry on.
    3. 
    
  3. A missing or malformed value only blows up at the moment the code first uses it, possibly hours after deploy.
    4. 
  


  

    The fix is to validate the entire environment once at startup,
    using Zod, and export a typed env object that the rest of the
    code reads from.
  


  
api/src/config/env.ts


const envSchema = z
  .object({
    NODE_ENV: z.enum(['development', 'test', 'production']).default('development'),
    PORT: z.coerce.number().int().positive().max(65_535).default(3000),
    LOG_LEVEL: z.enum(['fatal','error','warn','info','debug','trace','silent'])
                .default('info'),

    DATABASE_URL: z.string().url(),

    JWT_ACCESS_SECRET:  z.string().min(32, 'JWT_ACCESS_SECRET must be ≥32 chars'),
    JWT_REFRESH_SECRET: z.string().min(32, 'JWT_REFRESH_SECRET must be ≥32 chars'),
    JWT_ACCESS_TTL:  z.string().regex(/^\d+(ms|s|m|h|d)$/).default('15m'),
    JWT_REFRESH_TTL: z.string().regex(/^\d+(ms|s|m|h|d)$/).default('7d'),
    BCRYPT_ROUNDS:   z.coerce.number().int().min(10).max(15).default(12),

    CORS_ORIGINS: csvOrigins,
    RATE_LIMIT_WINDOW_MS: z.coerce.number().int().positive().default(60_000),
    RATE_LIMIT_MAX:       z.coerce.number().int().positive().default(100),
    JSON_BODY_LIMIT:      z.string().default('100kb'),
  })
  .refine((e) => e.JWT_ACCESS_SECRET !== e.JWT_REFRESH_SECRET, {
    message: 'JWT_ACCESS_SECRET and JWT_REFRESH_SECRET must differ',
    path: ['JWT_REFRESH_SECRET'],
  });

const parsed = envSchema.safeParse(process.env);

if (!parsed.success) {
  console.error('❌ Invalid environment variables:');
  for (const issue of parsed.error.issues) {
    console.error(`  • ${issue.path.join('.')}: ${issue.message}`);
  }
  process.exit(1);
}

export const env: Env = parsed.data;


  
What this gives us

  
    
    
  • 
      Types for free. type Env = z.infer<typeof envSchema>
      — the TypeScript type is derived from the schema. They cannot drift apart.
    
    • 
    
  • 
      Coercion. process.env.PORT is always a
      string, but z.coerce.number() converts it before validating
      the range.
    
    • 
    
  • 
      Sensible defaults. Most fields default; only secrets
      and DB URL are truly required.
    
    • 
    
  • 
      Cross-field rules. .refine() enforces that
      the access and refresh secrets are different — using the same
      secret would let a refresh token be misused as an access token.
    
    • 
    
  • 
      Fail-fast. If anything is wrong, the process prints
      every issue and exits before app.listen() is ever reached.
    
    • 
  


  

    Why fail-fast?
    A misconfigured server that starts is dangerous: it will accept
    traffic, then crash on the first request that touches the broken setting.
    A server that refuses to start is loud and obvious — the
    deployment fails, the rollback is automatic, no user ever sees a 500.

Prisma and the database

    
A schema file becomes types, queries, and migrations.

  


  

    The database is the durable heart of the application. Everything else
    (Express, React, JWTs) is rebuilt on every deploy; the database keeps
    living. Talking to PostgreSQL by hand — writing raw SQL with
    string concatenation — works but is unsafe and tedious. Prisma is
    an ORM (Object-Relational Mapper): you describe your
    tables in a schema file and Prisma generates a fully-typed query client.
  


  
The schema is the source of truth


  
api/prisma/schema.prisma (excerpt)


model User {
  id              String    @id @default(uuid(7)) @db.Uuid
  email           String    @unique @db.VarChar(254)
  passwordHash    String
  firstName       String    @db.VarChar(100)
  lastName        String    @db.VarChar(100)
  phone           String?   @db.VarChar(32)
  isActive        Boolean   @default(true)
  emailVerifiedAt DateTime?
  lastLoginAt     DateTime?
  createdAt       DateTime  @default(now())
  updatedAt       DateTime  @updatedAt
  deletedAt       DateTime?

  userRoles       UserRole[]
  refreshTokens   RefreshToken[]
  studentProfile  Student?
  teacherProfile  Teacher?

  @@index([deletedAt])
  @@map("users")
}


  

    Read the annotations: @id, @unique,
    @default, @db.VarChar(100). They translate
    directly into SQL constraints. @@map("users") says the table
    in PostgreSQL is named users (snake_case) while the Prisma
    model is User (PascalCase). Two conventions, one mapping
    table, no friction.
  


  
From schema to typed client



pnpm db:migrate    # generate + run a migration, then regen the client
pnpm db:generate   # just regen the typed client


  

    After this runs, your editor knows that prisma.user.findUnique()
    returns User | null, that where accepts only
    the columns you declared, and that create({ data }) requires
    every non-optional field. If you add a column to schema.prisma,
    your TypeScript code breaks at every usage site until you handle it.
  


  
The singleton trick


  

    PrismaClient owns a connection pool. Building two of them in the same
    process eats double the connections. In production we build exactly one;
    in dev with hot-reload (tsx watch), the module re-imports on
    every save and would build a fresh client each time, exhausting the pool
    in minutes. The fix is small and clever:
  


  
api/src/db/prisma.ts


const globalForPrisma = globalThis as unknown as { prismaInstance?: PrismaClient };

const buildPrismaClient = (): PrismaClient => {
  const isProd = process.env['NODE_ENV'] === 'production';
  return new PrismaClient({
    log: isProd ? ['warn', 'error'] : ['warn', 'error'],
    errorFormat: isProd ? 'minimal' : 'pretty',
  });
};

export const prisma: PrismaClient =
  globalForPrisma.prismaInstance ?? buildPrismaClient();

if (process.env['NODE_ENV'] !== 'production') {
  globalForPrisma.prismaInstance = prisma;
}


  

    In dev, the first import builds the client and stashes it on
    globalThis. Subsequent imports (after a hot-reload) find the
    stashed instance and reuse it. The pool stays at one connection.
  


  

    Why globalThis?
    Because module-scope variables get re-initialised when the module is
    re-imported. globalThis is shared across all imports of all
    versions of all modules in this process — it survives the
    re-import. In production where we never reload, the cache is bypassed.
  


  
A typical query, end to end


  
api/src/modules/auth/repository.ts


const userWithRolesInclude = {
  userRoles: { include: { role: true } },
} satisfies Prisma.UserInclude;

export const authRepository = {
  findUserByEmail: (email: string): Promise<UserWithRoles | null> => {
    return prisma.user.findUnique({
      where: { email: email.toLowerCase() },
      include: userWithRolesInclude,
    });
  },
  // ...
};


  

    Notice:
  

  
    
    
  • The repository function has a precise return type. Callers know exactly what they get.
    • 
    
  • email.toLowerCase() is applied at the query boundary. We treat email as case-insensitive without needing the citext PostgreSQL extension.
    • 
    
  • include pulls the user's roles in one query (a SQL join) rather than two trips to the DB — that is the classic N+1 query bug avoided.
    • 
    
  • satisfies Prisma.UserInclude verifies the include shape against Prisma's type without widening it. If you misspell userRoles, the compiler tells you.
    • 
  


  
Soft delete is a convention, not a feature


  

    Notice deletedAt DateTime? on most models. The application
    never truly deletes a user — it sets deletedAt = now().
    Repositories then filter out rows where deletedAt is not
    null when listing. This is "soft delete": preserved history,
    reversible mistakes, no orphaned foreign keys.
  


  

    Pitfall
    Soft delete is repository-enforced, not database-enforced. If you bypass
    the repository (e.g., write raw SQL elsewhere), you can accidentally show
    deleted users. The discipline is to keep all reads/writes for a
    model behind its repository.
  


  
Migrations: the schema is versioned


  

    When you change schema.prisma, you run
    pnpm db:migrate. Prisma diffs your schema against the
    database, writes a new SQL file under
    prisma/migrations/<timestamp>_<name>/, and
    applies it. That SQL file goes into git. Every developer and every
    environment ends up with the same database shape.
  


  

    Note
    Never edit a migration that has been merged. If you need to change
    something, write a new migration. Editing history is what makes
    deployments diverge.

The module pattern: route → controller → service → repository

    
One feature, four files, four jobs.

  


  

    This is the most important pattern in the server. Once you have it, the
    rest of api/ reads like prose. Every business module
    (auth/, users/, the planned
    classes/ and others) follows the same four-layer shape.
  


  

   HTTP request
        │
        ▼
   ┌──────────────────┐
   │  routes.ts       │   URL pattern + verb → controller fn
   │                  │   Plus per-route middleware (validate, rate-limit)
   └────────┬─────────┘
            ▼
   ┌──────────────────┐
   │  controller.ts   │   Read req, build "requester context",
   │                  │   call service, write response.
   │                  │   No business rules here.
   └────────┬─────────┘
            ▼
   ┌──────────────────┐
   │  service.ts      │   Authorisation. Business rules. Orchestration.
   │                  │   Calls one or more repository methods.
   │                  │   Throws AppError subclasses on policy violations.
   └────────┬─────────┘
            ▼
   ┌──────────────────┐
   │  repository.ts   │   Prisma queries. The only file that knows SQL.
   └──────────────────┘
  


  
routes.ts — the menu


  

    The routes file maps URLs to handlers and applies per-route middleware
    (validation, stricter rate limits, role gates). It contains no logic
    beyond wiring.
  


  
api/src/modules/users/routes.ts (excerpt)

controller.ts — translate, delegate, respond


  

    A controller is an adapter between Express and the service. It pulls
    typed data out of req, calls the service, and writes the
    response. If it grows past a dozen lines, the missing logic probably
    belongs in the service.
  


  
api/src/modules/users/controller.ts (excerpt)


const requireUser = (req: Request): RequesterContext => {
  if (req.user === undefined) {
    throw new UnauthorizedError();
  }
  return { id: req.user.id, roles: req.user.roles };
};

export const update: RequestHandler = async (req, res) => {
  const requester = requireUser(req);
  const params = req.params as unknown as UserIdParams;
  const user = await usersService.update(params.id, req.body as UpdateUserBody, requester);
  res.json(user);
};


  

    The controller never decides whether the update is allowed. It just
    forwards everything — including who is asking
    (requester) — to the service. That keeps controllers
    boring, which is what you want.
  


  
service.ts — the brain


  

    The service is the only layer where policy lives. Who may do
    what. What the rules say. Which repository calls happen and in what
    order. Errors thrown here are AppError subclasses with
    correct HTTP status codes (Chapter 12).

update: async (
  id: string,
  input: UpdateUserBody,
  requester: RequesterContext,
): Promise<PublicUser> => {
  const admin  = isAdmin(requester);
  const isSelf = requester.id === id;
  if (!admin && !isSelf) {
    throw new ForbiddenError();
  }

  // Self can patch firstName/lastName/phone. Email + isActive are admin-only.
  if (!admin && (input.email !== undefined || input.isActive !== undefined)) {
    throw new ForbiddenError('Only admins can change email or active status');
  }

  const data: Prisma.UserUpdateInput = {};
  if (input.firstName !== undefined) data.firstName = input.firstName;
  if (input.lastName  !== undefined) data.lastName  = input.lastName;
  if (input.phone     !== undefined) data.phone     = input.phone;
  if (admin && input.email     !== undefined) { /* ...uniqueness check... */ }
  if (admin && input.isActive  !== undefined) data.isActive = input.isActive;

  const updated = await usersRepository.update(id, data);
  if (updated === null) throw new NotFoundError('User');

  logger.info({ userId: id, requesterId: requester.id }, 'user updated');
  return toPublicUser(updated);
},


  

    Read it as policy, line by line. "Only admins or the user themselves
    may patch a user. Non-admins cannot change email or active status. Only
    set fields the client actually sent. Translate database result to public
    shape." Every rule is one or two lines, easy to test, easy to
    audit.
  


  

    Why is authorisation here, not in middleware?
    Some rules need context: "the requester equals the resource
    owner". A blanket requireRoles('ADMIN') middleware cannot
    express that — it would have to load the resource to know who owns
    it, which is the service's job anyway. So we put coarse role gates in
    middleware and fine-grained "admin or self" checks in the service.
  


  
repository.ts — the only file that knows about Prisma


  
api/src/modules/auth/repository.ts (excerpt)


export const authRepository = {
  findUserByEmail: (email: string) =>
    prisma.user.findUnique({
      where: { email: email.toLowerCase() },
      include: userWithRolesInclude,
    }),

  createUserWithRole: async (data, roleName) => {
    return prisma.$transaction(async (tx) => {
      const role = await tx.role.findUnique({ where: { name: roleName } });

    Two things to notice. First, repository functions are deliberately small
    — one query each. Second, when multiple writes must succeed or fail
    together (creating a user and attaching a role), we wrap them in
    prisma.$transaction. Either both rows are written or neither
    is.

Input validation with Zod

    
Refuse bad data at the door.

  


  

    A controller is allowed to trust the data it reads from
    req.body. That trust is bought with one piece of middleware:
    the Zod validator. Every endpoint declares the shape it expects, and the
    validator throws if the request doesn't fit.
  


  
The DTO file: schema and type in one place


  
api/src/modules/auth/dtos/auth.dto.ts


const emailSchema = z.string().trim().email().max(254).toLowerCase();
const passwordSchema = z.string().min(8).max(128);

export const registerBodySchema = z.object({
  email:     emailSchema,
  password:  passwordSchema,
  firstName: z.string().trim().min(1).max(100),
  lastName:  z.string().trim().min(1).max(100),
  phone:     z.string().trim().max(32).optional(),
}).strict();

export type RegisterBody = z.infer<typeof registerBodySchema>;


  

    Three things ride on this one declaration:
  


  
    
    
  1. 
      Runtime validation. An incoming body that has the
      wrong types, extra fields, or missing required values is rejected.
    
    2. 
    
  2. 
      Static type. z.infer turns the schema
      into a TypeScript type. The compiler then uses that type everywhere.
    
    3. 
    
  3. 
      Documentation. A junior engineer reading this file
      learns the API's accepted shape without opening a wiki.
    
    4. 
  


  
The .strict() modifier

  

    .strict() rejects any field the schema does not declare. A
    payload like { email, password, isAdmin: true } would not
    just have isAdmin stripped — the whole request is
    rejected. This blocks mass-assignment attacks where a
    client tries to set columns we never meant to expose.
  


  

    Pitfall
    Without .strict() (or its equivalent), Zod silently drops
    unknown keys. That seems forgiving until the day a client sends
    role: "ADMIN" alongside a valid registration payload and you
    forgot to ignore it on the way to prisma.user.create.
  


  
The validate middleware


  

    The middleware that applies a schema is small and reused everywhere:
  


  
api/src/shared/middleware/validate.ts


export const validate = (schemas: ValidationSchemas): RequestHandler => {
  return async (req, _res, next) => {
    try {
      if (schemas.body !== undefined) {
        req.body = await schemas.body.parseAsync(req.body);
      }
      if (schemas.query !== undefined) {
        const parsedQuery: unknown = await schemas.query.parseAsync(req.query);
        Object.defineProperty(req, 'query', { value: parsedQuery, ... });
      }
      if (schemas.params !== undefined) { /* ... */ }
      next();
    } catch (err) {
      if (err instanceof ZodError) {
        next(new ValidationError('Request validation failed', err.flatten()));
        return;
      }
      next(err);
    }
  };
};


  

    A few things to notice:
  


  
    
    
  • 
      The middleware replaces req.body with the parsed
      value. After validation, the request body has the exact, narrow type
      the schema declared — not a wider any.
    
    • 
    
  • 
      For req.query, Express 5 exposes it as a getter, so we use
      Object.defineProperty to shadow it. This is one of the
      few places you'll see that incantation.
    
    • 
    
  • 
      A ZodError is wrapped in our own ValidationError
      with the flattened issues attached. The central error handler turns it
      into a 400 with a structured body. (Chapter 12.)
    
    • 
  


  
Putting it together at the route


  
api/src/modules/auth/routes.ts


authRouter.post(

  validate({ body: registerBodySchema }),           // then parse
  controller.register,                              // then run
);


  

    By the time controller.register runs, the body has already
    been parsed, trimmed, lowercased (for email), and verified to have no
    extra keys. The controller can use req.body directly as
    RegisterBody.
  


  
Composing schemas


  

    Zod schemas are first-class values: you can build them up, reuse them,
    and refine them. The user DTOs show several patterns:
  



// Reuse a sub-schema across multiple DTOs
const emailSchema = z.string().trim().email().max(254).toLowerCase();

// .extend() — add fields
export const publicUserSchema = tokenUserSchema.extend({
  phone:     z.string().nullable(),
  isActive:  z.boolean(),
  createdAt: z.coerce.date(),
});

// .refine() — cross-field rules with a custom message
export const registerFormSchema = registerBaseSchema
  .extend({ confirmPassword: z.string().min(1) })

    message: 'Passwords do not match',

  });


  
The same idea in the client


  

    The browser has its own copy of the schemas in
    web/src/features/auth/schemas.ts. They mirror the server
    DTOs almost line for line. This is not duplication for its own sake
    — the client schemas drive form validation and the API client's
    response parsing. We will see this synchronisation in Chapter 20.
  


  

    Why validate the response too?
    Because the client cannot trust the server either. A deploy could
    ship a server change that broke the contract. With Zod on the response,

    before it shows up as an unrecoverable runtime crash.

Chapter 10 · Part II — The Server

    
Authentication: passwords, JWTs, and refresh-token rotation

    
"Who are you, and why should I believe you?"

  


  

    Authentication is the act of proving identity. We do it in two acts.
    First, the user logs in with email and password and we hand back two
    tokens. Second, for every subsequent request, the user presents the
    short-lived access token and we verify its signature without
    touching the database.
  


  
Act 1: Passwords


  

    We never store the user's password. We store a hash:
    a one-way transformation that takes seconds to verify but billions of
    years to reverse. The function we use is bcrypt, configured
    with a "rounds" parameter that controls how expensive a single hash is.
  


  
api/src/shared/utils/password.ts


import bcrypt from 'bcryptjs';
import { env } from '../../config/env.js';

export const hashPassword = (plain: string): Promise<string> =>
  bcrypt.hash(plain, env.BCRYPT_ROUNDS);

export const verifyPassword = (plain: string, hash: string): Promise<boolean> =>
  bcrypt.compare(plain, hash);


  

    The hash includes the algorithm, cost, and a per-user random
    salt: $2a$12$Q7p.... The salt means two users with
    the same password get different hashes — an attacker who steals
    the database can't crack the whole user table at once.
  


  

    Why bcrypt, not SHA-256?
    SHA-256 is fast. Bcrypt is deliberately slow — about
    ~250ms for 12 rounds. That cost is invisible to a real user logging in
    once, but devastating to an attacker who wants to test a billion guesses
    against a stolen hash file.
  


  
The constant-time trick at login


  

    Compare a real login (user exists, wrong password) to a fake one (user
    doesn't exist). A naive server skips the hash check when the user
    doesn't exist, so the fake response arrives faster — revealing
    that the email is not registered. This is a timing oracle.
    The fix is to spend the same wall-clock time in either case:
  


  
api/src/modules/auth/service.ts


// Pre-computed once at module load.
const TIMING_DUMMY_HASH = bcrypt.hashSync('timing_equalization_only', env.BCRYPT_ROUNDS);

login: async (input, context) => {
  const user = await authRepository.findUserByEmail(input.email);

  const fail = (): never => { throw new UnauthorizedError('Invalid credentials'); };

  if (user === null || user.deletedAt !== null || !user.isActive) {
    if (user === null) {
      await verifyPassword(input.password, TIMING_DUMMY_HASH);  // burn the cycles
    }
    fail();
  }

  const valid = await verifyPassword(input.password, user.passwordHash);
  if (!valid) fail();
  // ...issue tokens
}


  

    Notice also: every failure throws the same error message,
    "Invalid credentials". We never tell the client whether the
    email was unknown, the password was wrong, or the account is disabled.
    Each piece of information is a clue an attacker can use.
  


  
Act 2: JWTs (access tokens)


  

    A JWT (JSON Web Token) is a tamper-evident envelope of
    JSON. It looks like three base64 chunks joined by dots:
    header.payload.signature. The signature is an HMAC over
    the header+payload using a secret only the server knows. If a client
    edits the payload, the signature no longer matches, and verification
    fails.
  


  
api/src/shared/utils/jwt.ts


export const signAccessToken = (input: { sub, email, roles }): string => {
  const claims = { ...input, type: 'access', jti: uuidv7() };
  return jwt.sign(claims, env.JWT_ACCESS_SECRET, {
    expiresIn: env.JWT_ACCESS_TTL,
    issuer:    'school-mngt-api',
    audience:  'school-mngt-api',
    algorithm: 'HS256',
  });
};

export const verifyAccessToken = (token: string): AccessTokenClaims => {
  try {
    const decoded = jwt.verify(token, env.JWT_ACCESS_SECRET, {
      issuer: 'school-mngt-api', audience: 'school-mngt-api', algorithms: ['HS256'],
    });
    if (typeof decoded === 'string' || decoded.type !== 'access') {
      throw new UnauthorizedError('Invalid token type');
    }
    return decoded as AccessTokenClaims;
  } catch (err) {
    if (err instanceof jwt.TokenExpiredError)  throw new UnauthorizedError('Access token expired');
    if (err instanceof jwt.JsonWebTokenError)  throw new UnauthorizedError('Invalid access token');
    throw err;
  }
};


  

    A few choices to note. The access token TTL is 15 minutes (short). The
    jti (JWT ID) is a unique UUIDv7 per token — useful if
    we ever want a deny-list. The issuer and audience
    claims make sure tokens issued by other services in the same network
    cannot be reused here.
  


  
Why two tokens?


  

    The access token is stateless. To revoke a stateless token, you would
    need a deny-list and a database hit on every request — defeating
    the purpose. So we keep the access token short-lived (15 min) and
    add a stateful refresh token (7 days). The refresh
    token is tracked in the database and can be revoked instantly.

Refresh-token rotation and reuse detection


  

    A naive design hands the same refresh token back over and over. If it
    leaks, an attacker can refresh indefinitely. We do something cleverer:
    each refresh issues a brand-new refresh token and revokes the
    old one. A refresh token is therefore single-use.
  


  

    Now consider what happens if an attacker steals a refresh token and
    the legitimate user uses it once. One of them will eventually try the
    already-revoked old token. At that moment we know: someone is presenting
    a token that should no longer exist. We can't tell whether it's the
    attacker or the victim — but we don't have to. We revoke
    every active session for that user and force re-login.
  


  
api/src/modules/auth/service.ts


refresh: async (rawToken: string, context: RequestContext): Promise<TokenPair> => {
  const tokenHash = hashToken(rawToken);
  const stored = await authRepository.findRefreshTokenByHash(tokenHash);

  if (stored === null) throw new UnauthorizedError('Invalid refresh token');
  if (stored.expiresAt.getTime() < Date.now())
    throw new UnauthorizedError('Refresh token expired');

  if (stored.revokedAt !== null) {
    // Reuse detected (RFC 6819 §5.2.2.3).
    const revokedCount = await authRepository.revokeAllUserRefreshTokens(stored.userId);
    logger.warn({ userId: stored.userId, revokedCount },
      'refresh-token reuse detected — all sessions revoked');
    throw new UnauthorizedError('Refresh token reused — session revoked');
  }

  const user = await authRepository.findUserById(stored.userId);
  if (user === null || user.deletedAt !== null || !user.isActive)
    throw new UnauthorizedError('User unavailable');

  const pair = await issueTokens(user, context);
  await authRepository.revokeRefreshToken(stored.id, pair.refreshTokenId);
  return pair;
},


  
What we store, what we don't


  

    We never store the raw refresh token. We store a SHA-256 hash of it. A
    leak of the database leaves an attacker with no usable refresh tokens.
  


  
api/src/shared/utils/token.ts


const REFRESH_TOKEN_BYTES = 48;

export const generateRefreshToken = (): string =>
  randomBytes(REFRESH_TOKEN_BYTES).toString('base64url');

export const hashToken = (raw: string): string =>
  createHash('sha256').update(raw).digest('hex');


  

    Why SHA-256 and not bcrypt? Refresh tokens are already 48 bytes of
    high-entropy randomness — nobody can guess them, so the cost of
    bcrypt buys nothing. A fast hash is enough.
  


  
The auth middleware


  

    On every authenticated request, this middleware reads the
    Authorization: Bearer <token> header, verifies the
    JWT, and attaches the user's identity to req.user so
    downstream handlers can use it.
  


  
api/src/modules/auth/middleware.ts


const BEARER = 'Bearer ';

export const requireAuth: RequestHandler = (req, _res, next) => {
  const auth = req.header('authorization');
  if (auth === undefined || !auth.startsWith(BEARER)) {
    next(new UnauthorizedError('Missing or malformed Authorization header'));
    return;
  }
  const token = auth.slice(BEARER.length).trim();
  if (token.length === 0) {
    next(new UnauthorizedError('Empty bearer token'));
    return;
  }
  try {
    const claims = verifyAccessToken(token);
    req.user = { id: claims.sub, email: claims.email, roles: claims.roles };
    next();
  } catch (err) {
    next(err);
  }
};


  

    A successful pass attaches a typed req.user — the rest
    of the request gets to assume it's there. (The TypeScript declaration for
    req.user lives in shared/types/express.d.ts.)
  


  
Rate limits at the auth boundary


  

    Authentication endpoints are attractive to brute-force attackers, so they
    get stricter limits than the global one:
  


  
api/src/modules/auth/routes.ts


const loginLimiter    = rateLimit({ windowMs: 15*60*1000, limit: 5  });
const registerLimiter = rateLimit({ windowMs: 60*60*1000, limit: 3  });
const refreshLimiter  = rateLimit({ windowMs:    60*1000, limit: 30 });


  

    Five logins per fifteen minutes per IP. A human getting their password
    wrong four times is fine; an attacker trying a million passwords gets
    nowhere.
  


  

    Pitfall
    Never roll your own crypto, your own JWT verifier, or your own
    password-hashing scheme. Use library functions written by people who
    have spent a career thinking about timing attacks, padding oracles, and
    constant-time comparison. bcrypt.compare and
    jwt.verify already do the right thing.

Authorisation: roles, middleware, and the service layer

    
"You're logged in. But may you do this?"

  


  

    Authentication answers who. Authorisation answers may they.
    The two are easy to confuse but very different in code: authentication is
    one check at the edge; authorisation runs on every request, often with
    knowledge of the specific resource being touched.
  


  
Roles


  

    The project uses role-based access control (RBAC).
    Every user holds zero or more roles from a fixed set:
  


  
api/prisma/schema.prisma


enum RoleName {
  ADMIN
  TEACHER
  STUDENT
  PARENT
  STAFF
}


  

    The link is many-to-many: one user can be both teacher and
    parent. Roles are stored in a join table (user_roles)
    and travel with the access token — the JWT payload includes the
    user's roles, so authorisation does not need a database hit.
  


  
Two layers of gate


  

    Authorisation in this project happens at two levels: coarse
    gates in middleware (whole routes), fine gates in the service
    (resource-aware).
  


  
Coarse gate: middleware


  
api/src/modules/auth/middleware.ts


export const requireRoles = (...allowed: readonly RoleName[]): RequestHandler => {
  if (allowed.length === 0) {
    throw new Error('requireRoles: at least one role must be supplied');
  }
  return (req, _res, next) => {
    if (req.user === undefined) { next(new UnauthorizedError()); return; }
    const matched = req.user.roles.some((r) => allowed.includes(r));
    if (!matched) {
      next(new ForbiddenError(`Requires role: ${allowed.join(' or ')}`));
      return;
    }
    next();
  };
};


  

    Use it on routes where the rule has no nuance: "only admins can list
    all users", "only teachers and admins can mark attendance".
    Pair it with requireAuth upstream.
  


  
Fine gate: the service


  

    Coarse gates can't express "admin or self". For that, the service reads
    the requester context — the controller's
    { id, roles } — alongside the resource:
  


  
api/src/modules/users/service.ts


const isAdmin = (r: RequesterContext): boolean => r.roles.includes(RoleName.ADMIN);

const requireAdminOrSelf = (r: RequesterContext, targetId: string): void => {
  if (!isAdmin(r) && r.id !== targetId) {
    throw new ForbiddenError();
  }
};

get: async (id: string, requester: RequesterContext): Promise<PublicUser> => {
  requireAdminOrSelf(requester, id);
  const user = await usersRepository.findById(id);
  if (user === null) throw new NotFoundError('User');
  return toPublicUser(user);
},


  

    The same rule applies to password changes, profile edits, and other
    self-service actions. Notice the rule is a tiny helper, named exactly
    after what it enforces. Someone reading get for the first
    time learns the policy from one English sentence: require admin or
    self.
  


  
Defence against the "can-I-delete-myself" class of bug


  

    Some rules don't fit neatly into "admin or self": an admin
    must not delete themselves; an admin must not remove their own
    ADMIN role. These are guard rails to prevent a user from locking
    themselves out of the system. They live next to the operation they
    protect:
  



delete: async (id, requester) => {
  requireAdmin(requester);
  if (id === requester.id) throw new ConflictError('Cannot delete yourself');
  // ...
},

removeRole: async (id, role, requester) => {
  requireAdmin(requester);
  if (id === requester.id && role === RoleName.ADMIN) {
    throw new ConflictError('Cannot remove your own ADMIN role');
  }
  // ...
},


  
Side effects of revocation


  

    When an admin deletes a user (soft-delete) or changes a password, we
    revoke every active refresh token for that user. The deleted
    user's other browser tabs cannot stay logged in:
  



await usersRepository.softDelete(id);
await authRepository.revokeAllUserRefreshTokens(id);   // log them out everywhere


  

    The same call happens on password change. Anyone who knew the old
    password and was already logged in is now kicked out.
  


  

    Why two layers?
    Middleware is good at coarse, declarative rules — you see them in
    routes.ts at a glance. Service-level checks have access to
    the loaded resource and the requester together — they can express
    the rules middleware cannot. The two complement each other.
  


  

    Pitfall
    Never authorise only in the client by hiding a button. A user
    with dev-tools can dispatch the request anyway. Hide the button for UX,
    enforce the rule in the service for safety. The server is the only
    place authorisation actually happens.

Errors: one taxonomy, one handler

    
Throw freely; the boundary makes it consistent.

  


  

    Inconsistent error responses are a UX disaster. Each endpoint deciding
    its own format means each client widget needs its own parser. This
    project takes the opposite approach: every error becomes a typed
    AppError, and one central middleware turns it into a
    consistent JSON body with a stable status code.
  


  
The taxonomy


  
api/src/shared/errors/app-error.ts


export class AppError extends Error {
  public readonly statusCode: number;
  public readonly code: string;
  public readonly details: unknown;
  public readonly isOperational: boolean;

  constructor(message, statusCode, code, details = null, isOperational = true) {
    super(message);
    this.name = this.constructor.name;
    this.statusCode = statusCode;
    this.code = code;
    this.details = details;
    this.isOperational = isOperational;
    Error.captureStackTrace?.(this, this.constructor);
  }
}

export class ValidationError       extends AppError { /* 400 VALIDATION_ERROR */ }
export class UnauthorizedError     extends AppError { /* 401 UNAUTHORIZED    */ }
export class ForbiddenError        extends AppError { /* 403 FORBIDDEN       */ }
export class NotFoundError         extends AppError { /* 404 NOT_FOUND       */ }
export class ConflictError         extends AppError { /* 409 CONFLICT        */ }
export class UnprocessableEntityError extends AppError { /* 422              */ }
export class TooManyRequestsError  extends AppError { /* 429 RATE_LIMITED    */ }
export class InternalError         extends AppError { /* 500 (not operational) */ }


  

    Four fields per error:
  


  

    
statusCode
HTTP status. The central handler reads this directly — no res.status(400) scattered across the codebase.

    
code
A stable machine-readable string. Clients switch on this, not on the human message. VALIDATION_ERROR is forever; "Password too short" is a copy change away.

    
details
Optional structured payload — field-level Zod errors, for instance.

    
isOperational
True for errors we expected and handled (validation, auth, not-found). False for bugs — an InternalError represents "the system did something it shouldn't have". Different log severity, different alerting.

  


  
How services throw

  

    Service code throws these directly. It does not call res.status.
    It does not log. It throws and lets the boundary handle it.
  



if (existing !== null)         throw new ConflictError('Email already registered');
if (!isAdmin(requester))       throw new ForbiddenError('Admin role required');
if (user === null)             throw new NotFoundError('User');
if (input.password === wrong)  throw new UnauthorizedError('Invalid credentials');


  
The boundary: one error handler

const normalize = (err: unknown): AppError => {
  if (err instanceof AppError) return err;
  if (err instanceof ZodError) return new ValidationError('Request validation failed', err.flatten());
  if (err instanceof Prisma.PrismaClientKnownRequestError)   return normalizePrismaError(err);
  if (err instanceof Prisma.PrismaClientValidationError)     return new ValidationError('Database query validation failed');
  if (err instanceof Error)    return new InternalError(err.message);
  return new InternalError('Unknown error');
};

export const errorHandler: ErrorRequestHandler = (err, req, res, _next) => {
  const normalized = normalize(err);
  const isServerError = normalized.statusCode >= 500;

  const logPayload = {
    err, requestId: req.id, statusCode: normalized.statusCode,
    code: normalized.code, method: req.method, path: req.originalUrl,
    operational: normalized.isOperational,
  };

  if (isServerError || !normalized.isOperational) {
    logger.error(logPayload, normalized.message);
  } else {
    logger.warn(logPayload, normalized.message);
  }

  const isProd = env.NODE_ENV === 'production';
  const body = {
    error: {
      code: normalized.code,
      message: isServerError && isProd ? 'Internal server error' : normalized.message,
      requestId: req.id,
    },
  };
  if (!isProd && normalized.stack !== undefined) body.error.stack = normalized.stack;
  if (normalized.details && !(isServerError && isProd)) body.error.details = normalized.details;

  res.status(normalized.statusCode).json(body);
};

  1. 
      Normalise. Whatever was thrown becomes an
      AppError. Zod errors become ValidationError.
      Prisma errors become ConflictError (for P2002 unique
      violations), NotFoundError (P2025), or
      InternalError. Unknown errors become InternalError.
    
    2. 
    
  2. 
      Log. Server errors and non-operational errors get
      logger.error; everything else (validation, auth, etc.)
      gets logger.warn. The original error object is included so
      stack traces survive into the log.
    
    3. 
    
  3. 
      Respond. Always the same JSON shape. In production,
      5xx messages are scrubbed to "Internal server error" —
      we never leak internal details to the client. In development, the
      stack is included to help you debug.
    
    4.

    "requestId": "0193e..."

Translating Prisma errors



const normalizePrismaError = (err: Prisma.PrismaClientKnownRequestError): AppError => {
  switch (err.code) {
    case 'P2002': {                                    // unique constraint
      const target = err.meta?.target;
      return new ConflictError('Unique constraint violation', { target });
    }
    case 'P2003': return new ConflictError('Foreign key constraint violation');
    case 'P2025': return new NotFoundError('Resource');
    default:      return new InternalError('Database error', { prismaCode: err.code });
  }
};


  

    Prisma's internal error codes are useful but they speak in Prisma terms.
    The boundary translates them into our own taxonomy — service code
    upstream can race a unique-constraint violation and trust that it will
    be served back as a clean 409, not a confusing 500.
  


  

    Why centralise this?
    Imagine the alternative: every endpoint catches its own errors and
    decides its own response. Twenty endpoints, twenty subtly different
    shapes. The first time you forget a try/catch, the user gets a stack
    trace. The boundary pattern means you cannot forget.
  


  
The async-error gotcha (and why we don't have it)


  

    In Express 4, an async route that threw would crash the process unless
    you wrapped it manually. Express 5 (used here) fixed this:
    async functions whose promises reject are forwarded to
    next(err) automatically. That is why every controller
    can be a plain async function with no try/catch.
  


  

    Pitfall
    Never console.log(err) in a controller and then send the
    error to the client unchanged. You will eventually leak a database
    password (it really happens). Use the central handler.

If you can't see it, you can't fix it.

  


  

    A server that emits no logs is invisible. The first time a user reports

    accept that an instance with a broken DB will keep serving 500s. Two
    endpoints let you do the right thing for both questions.

The provider stack

  return (

        <AuthBootstrap>
          <RouterProvider router={router} />
        </AuthBootstrap>
        <ToastViewport />
      </QueryClientProvider>
    </ErrorBoundary>

    The order is deliberate. Read it from outside in:
  


  
    
    
  1. 
      ErrorBoundary — catches any uncaught render error
      from anything inside it. If the app explodes, this is the last line of
      defence; the user sees a friendly "something went wrong" panel, not a
      blank page.
    
    2. 
    
  2. 
      QueryClientProvider — makes a single TanStack
      Query cache available to every descendant. Without this, hooks like
      useQuery have nowhere to store their data.
    
    3. 
    
  3. 
      AuthBootstrap — resolves the session before the
      router renders. It checks localStorage for a refresh
      token, tries to exchange it for an access token, and only then lets
      the rest of the tree mount. Chapter 19 explains why this matters.
    
    4. 
    
  4. 
      RouterProvider — owns the URL-to-component
      mapping. Once we're inside this, route loaders and components can
      assume auth is resolved.
    
    5. 
    
  5. 
      ToastViewport — rendered as a sibling of the
      router, so it survives navigation. It reads from the toast store and
      portals notifications into the corner of the screen.
    
    6.

    Every provider depends on the ones outside it. QueryClient
    needs ErrorBoundary above to catch its surprises;
    AuthBootstrap needs QueryClient to invalidate
    user-scoped caches on logout; RouterProvider needs auth
    resolved so loaders can make routing decisions. Swapping any two would
    break something specific.

A typed API client with auto-refresh

    
Every request, the same shape, the same trust.

  


  

    Sprinkling fetch() across the codebase is the first thing
    every junior frontend engineer does. Then come the duplicated headers,
    the inconsistent error handling, the forgotten JSON.stringify,
    the missing access-token refresh. The fix is one wrapper that every
    feature service calls.
  


  
The signature


  
web/src/lib/api-client.ts


export interface RequestConfig<T> {
  path: string;
  method?: 'GET' | 'POST' | 'PUT' | 'PATCH' | 'DELETE';
  body?: unknown;
  query?: Record<string, QueryValue>;
  headers?: Record<string, string>;
  signal?: AbortSignal;
  schema: ZodType<T>;        // required — validates the response
  skipAuth?: boolean;          // for /login, /refresh, /register
}

export const apiClient = <T>(config: RequestConfig<T>): Promise<T> => { ... };


  

    The required schema field is the trick.
    Every caller hands in a Zod schema that describes the response. The
    client parses the JSON through that schema and returns the parsed,
    fully-typed value. Callers never deal with unknown.
  


  

    Service code becomes pleasingly compact:

export const authService = {
  login: async (input: LoginFormValues): Promise<TokenPair> => {
    const pair = await apiClient({
      path: '/auth/login',
      method: 'POST',
      body: input,
      schema: tokenPairSchema,
      skipAuth: true,
    });
    persistSession(pair);
    return pair;
  },

  me: async (): Promise<PublicUser> => apiClient({
    path: '/auth/me',
    method: 'GET',
    schema: publicUserSchema,
  }),
};


  
What the wrapper does for you


  
    
    
  1. 
      Builds the URL: prefix with VITE_API_BASE_URL, append
      query params, encode safely.
    
    2. 
    
  2. 
      Sets headers: Content-Type: application/json, and
      Authorization: Bearer <accessToken> unless
      skipAuth.
    
    3. 
    
  3. 
      Serialises the body to JSON.
    
    4. 
    
  4. 
      On a 401, transparently tries to refresh the access token and retries
      the request. Once.
    
    5. 
    
  5. 
      Parses the response through the supplied Zod schema. If the server
      drifted from the contract, you get a clear ZodError, not a runtime
      surprise five components deep.
    
    6. 
    
  6. 
      On non-2xx, throws a typed ApiError carrying the server's
      error.code and error.message.
    
    7. 
  


  
The refresh dance


  

    Imagine the user opens the app after their access token has expired.
    Five components mount simultaneously and each fires a request. All five
    get back 401. A naive implementation would now call /refresh
    five times, race them, and confuse the rotation logic on the server.
  


  

    We dedupe by storing the in-flight refresh promise on a module-level
    variable. The first caller starts the refresh; the others
    await the same promise.
  


  
web/src/lib/api-client.ts (excerpt)


let refreshPromise: Promise<string | null> | null = null;

const performRefresh = async (): Promise<string | null> => {
  const refreshToken = authStorage.getRefreshToken();
  if (refreshToken === null) return null;

  const response = await fetch(buildUrl('/auth/refresh', undefined), {
    method: 'POST',
    headers: { 'Content-Type': 'application/json' },
    body: JSON.stringify({ refreshToken }),
  });
  if (!response.ok) return null;

  const result = tokenPairSchema.safeParse(await response.json());
  if (!result.success) return null;

  authStorage.setRefreshToken(result.data.refreshToken);
  useAuthStore.getState().setAccessToken(result.data.accessToken);
  return result.data.accessToken;
};

export const refreshAccessToken = async (): Promise<string | null> => {
  if (refreshPromise === null) {
    refreshPromise = performRefresh().finally(() => { refreshPromise = null; });
  }
  return refreshPromise;
};


  

    The retry inside the main client function is the simple half: if a
    request comes back 401 and we haven't already retried, call
    refreshAccessToken(), wait for the new token, retry the
    original request once. If that fails, give up and surface the
    error.
  


  

    Why one retry?
    Retrying forever would mean an unrecoverably-broken session loops the
    page into oblivion. Retrying once handles the common case (token
    expired naturally) without masking a real auth failure.
  


  
Storage of the refresh token


  

    The refresh token sits in localStorage via a small
    authStorage module that nobody else imports. The access
    token lives only in memory (the Zustand store) and is regenerated by
    AuthBootstrap on every page load.
  


  

    Pitfall
    localStorage is readable by any JavaScript on the same
    origin, so an XSS bug can exfiltrate the refresh token. For
    higher-security apps, you would put the refresh token in an
    HttpOnly cookie that JavaScript cannot read. The trade-off in this
    project is simplicity vs. that hardening.

    Anything non-2xx becomes an ApiError instance:

Zustand — a tiny, hookless store

    selector hook; only re-renders happen when the selected slice changes.
  


  
web/src/store/auth-store.ts


export type AuthStatus = 'loading' | 'authenticated' | 'unauthenticated';

interface AuthState {
  status: AuthStatus;
  user: TokenUser | null;
  accessToken: string | null;
  setSession: (user: TokenUser, accessToken: string) => void;
  setAccessToken: (accessToken: string) => void;
  clearSession: () => void;
  setStatus: (status: AuthStatus) => void;
}

export const useAuthStore = create<AuthState>((set) => ({
  status: 'loading',
  user: null,
  accessToken: null,
  setSession: (user, accessToken) => { set({ status: 'authenticated', user, accessToken }); },
  setAccessToken: (accessToken) => { set({ accessToken }); },
  clearSession: () => { set({ status: 'unauthenticated', user: null, accessToken: null }); },
  setStatus: (status) => { set({ status }); },
}));


  

    Three usage patterns:
  


  
web/src/features/auth/hooks/useAuth.ts


// In a component — subscribe to a slice

// From non-React code — mutate

    The lack of a provider is intentional. Any module in any tree can read
    or write the store. That is exactly what the API client needs — it
    is not a React component, but it needs the current access token and the
    ability to clear the session.

TanStack Query — the server cache


  

    The list of users is on the server, not in your component. But if every
    component re-fetched on every render, you would have an unusable app.
    What you want is: "hand me the latest data; refetch if it's stale;
    dedupe with other components that want the same data; throw away cached
    data on logout." That is what TanStack Query does.
  


  
web/src/features/users/hooks.ts (sketch)


export const useUsers = (query: ListUsersQuery) =>
  useQuery({
    queryKey: ['users', query],
    queryFn:  () => usersService.list(query),
  });

  useQuery({
    queryKey: ['users', 'detail', id],
    queryFn:  () => usersService.get(id),
  });

export const useCreateUser = () => {
  const qc = useQueryClient();
  return useMutation({
    mutationFn: usersService.create,
    onSuccess: () => qc.invalidateQueries({ queryKey: ['users'] }),
  });
};


  

    
queryKey
A stable identifier for "the same data". Components that supply the same key see the same cache entry. Add a parameter to the key when the data depends on it.

    
queryFn
How to fetch the data. Usually a call to a service function we wrote in Chapter 16.

    
invalidateQueries
"Mark this cache key (and anything starting with it) as stale, refetch if visible." Used after mutations to keep lists in sync with details.

  


  
Clearing the cache on logout


  
web/src/features/auth/hooks/useLogout.ts


export const useLogout = (options = {}): UseMutationResult<void, ApiError, void> => {
  const navigate = useNavigate();
  const queryClient = useQueryClient();
  const redirectTo = options.redirectTo ?? '/login';

  return useMutation({
    mutationFn: authService.logout,
    onSettled: async () => {
      queryClient.clear();              // critical
      await navigate(redirectTo, { replace: true });
    },
  });
};


  

    Why queryClient.clear()?
    Cached data may be user-scoped. If a second user logs in on the same
    browser, they should not see the first user's data, even for a flicker.
    Clearing the cache on logout is a privacy invariant, not a perf
    detail.
  


  
The mental model


  

    The rule of thumb: if the data has a server endpoint, use TanStack
    Query. If it doesn't, use Zustand. Trying to put server data in
    Zustand means re-implementing caching, invalidation, dedupe, retry by
    hand. Trying to put pure UI state in TanStack Query is awkward because
    there is no queryFn.
  


  

    Pitfall
    Do not store the same value in both Zustand and TanStack Query.
    They will drift. The user profile lives in Zustand (because the API
    client needs it imperatively); cached query results stay in TanStack
    Query. Two stores, no overlap.

    Forms in raw React are surprisingly hard. You manage one useState
    per field, one for each error, one for "is submitting", one for "is dirty".
    You debounce, you focus, you reset. Two libraries collapse all of that
    into a few lines: react-hook-form for the state
    machinery, Zod (via the resolver adapter) for validation.
  


  
The LoginForm, in full


  
web/src/features/auth/components/LoginForm.tsx


export const LoginForm = ({ onSuccess }: LoginFormProps): ReactElement => {
  const login = useLogin();
  const {
    register,
    handleSubmit,
    formState: { errors, isSubmitting },
  } = useForm<LoginFormValues>({
    resolver: zodResolver(loginFormSchema),
    defaultValues: { email: '', password: '' },
    mode: 'onSubmit',
  });

  const onSubmit = handleSubmit((data) => {
    login.mutate(data, {
      onSuccess: () => {
        toast.success('Signed in');
        onSuccess();
      },
      onError: (err) => {
        toast.error(errorMessage(err));
      },
    });
  });

  const isPending = login.isPending || isSubmitting;

  return (
    <form onSubmit={onSubmit} noValidate>
      <FormField label="Email" htmlFor="login-email" required error={errors.email?.message}>
        <Input id="login-email" type="email" autoComplete="email" autoFocus
               invalid={errors.email !== undefined}
               {...register('email')} />
      </FormField>
      <FormField label="Password" htmlFor="login-password" required error={errors.password?.message}>
        <PasswordInput id="login-password" autoComplete="current-password"
                       invalid={errors.password !== undefined}
                       {...register('password')} />
      </FormField>
      <Button type="submit" fullWidth size="lg" isLoading={isPending} loadingText="Signing in…">
        Sign in
      </Button>
    </form>
  );
};


  

    Read the moving parts:
  


  

    
useForm
Initialises the form. resolver wires Zod in; defaultValues set the starting state; mode chooses when to validate (onSubmit, onChange, onBlur).

    
register
Returns the props for an input: name, onChange, onBlur, ref. Spread them into your input and the field is wired.

    
handleSubmit
Wraps your submit handler. Runs validation; on success, hands you the typed, parsed values; on failure, populates errors and refuses to call your handler.

    
formState.errors
One entry per field with a validation problem. We render it via the FormField component below the input.

    
formState.isSubmitting
True while your async submit handler is in flight. Combined with the mutation's isPending to drive the button's loading state.

Auth bootstrap and the loading triangle

    
Three states to consider, not two.

  


  

    The first time a page loads, the app does not yet know whether the user
    is signed in. There is a refresh token in localStorage or
    there isn't; either way we have to try to exchange it for an access
    token before we can answer. During that exchange, neither
    authenticated nor unauthenticated is correct.
  


  

    Hence the third state: loading. The Zustand store starts
    here, and a single bootstrap component flips it to a definite answer
    before the router renders.
  


  

   ┌──────────────┐
   │   loading    │ ← initial; AuthBootstrap is deciding
   └──┬───────────┘
      │  refresh succeeded
      ▼
   ┌──────────────────┐
   │ authenticated    │ ← router can show protected routes
   └──┬───────────────┘
      │  logout / refresh failed
      ▼
   ┌──────────────────┐
   │ unauthenticated  │ ← router redirects to /login
   └──────────────────┘
  


  
The component


  
web/src/features/auth/components/AuthBootstrap.tsx


export const AuthBootstrap = ({ children }: Props): ReactElement => {
  const status = useAuthStore((s) => s.status);
  const [didRun, setDidRun] = useState(false);

  useEffect(() => {
    if (didRun) return;
    setDidRun(true);

    const run = async (): Promise<void> => {
      const refreshToken = authStorage.getRefreshToken();
      if (refreshToken === null) {
        useAuthStore.getState().clearSession();
        return;
      }
      const newAccessToken = await refreshAccessToken();
      if (newAccessToken === null) {
        useAuthStore.getState().clearSession();
      }
    };
    void run();
  }, [didRun]);

  if (status === 'loading') {
    return (
      <div style={{ display: 'grid', placeItems: 'center', minHeight: '100vh' }}>
        <Spinner label="Loading session" />
      </div>
    );
  }
  return <>{children}</>;
};


  

    Walking through it:
  


  
    
    
  1. On mount, check localStorage for a refresh token.
    2. 
    
  2. If there isn't one, jump straight to unauthenticated.
    3. 
    
  3. If there is, call refreshAccessToken() — that
        function (Chapter 16) returns either a new token (also setting
        the access token + user in the store via the refresh response) or
        null on failure. On failure, clear the session.
    4. 
    
  4. While all of the above is in flight, render a spinner. Crucially,
        the router does not mount during this window.
    5. 
  


  
Why the router must wait


  

    Imagine the router mounting while status === 'loading'. The
    requireAuthLoader would see the user as not authenticated
    and immediately redirect to /login — even though the
    refresh would succeed two hundred milliseconds later. The user sees a
    flash of the login page before being bounced back. The spinner gate
    prevents that flicker.
  


  

    Why the didRun ref?
    React's StrictMode double-invokes effects in development. Without the
    guard, the bootstrap would run twice and fire two refresh requests,
    which the server's reuse detection would then revoke. A boolean flag
    makes the effect idempotent.
  


  
The post-login path


  

    After a successful login (or register), authService.login
    calls persistSession(pair):
  



const persistSession = (pair: TokenPair): void => {
  authStorage.setRefreshToken(pair.refreshToken);
  useAuthStore.getState().setSession(pair.user, pair.accessToken);
};


  

    Refresh token to localStorage; user + access token to the
    store; status flips to authenticated; the loader
    that redirects authenticated users away from /login
    kicks in.
  


  
And on logout


  

    The logout mutation hits POST /auth/logout on the server,
    then unconditionally clears the local session and TanStack Query cache.
    Even if the server call fails, the user is locally logged out:
  



logout: async (): Promise<void> => {
  const refreshToken = authStorage.getRefreshToken();
  try {
    if (refreshToken !== null) {
      await apiClient({ path: '/auth/logout', method: 'POST',
                        body: { refreshToken }, schema: z.void(), skipAuth: true });
    }
  } finally {
    authStorage.clearRefreshToken();
    useAuthStore.getState().clearSession();
  }
},


  

    The finally is the safety net. A user who clicks "Sign out"
    while their wifi drops should not be stranded in a half-logged-in state.
  


  

    Pitfall
    Don't put bootstrap logic in many components. There must be exactly
    one place that decides "loading vs authenticated vs unauthenticated".
    Spreading the decision is how you end up with race conditions and
    flashes of the wrong UI.

Shared shapes: keeping client and server honest

    
Two codebases, one contract.

  


  

    The client and server are separate projects with separate
    package.json files. They cannot directly import each other's
    types — nor should they; deploying them together couples concerns
    that ought to stay independent. So how do we make sure they agree on
    what a User looks like?
  


  

    The answer in this project is parallel Zod schemas.
    Both sides write a schema for each shared shape; both sides derive their
    TypeScript types from those schemas. The schemas mirror each other.
    Drift is caught the moment a request fails parsing.
  


  
Side-by-side


  
api/src/modules/auth/dtos/auth.dto.ts (server)


export const registerBodySchema = z.object({
  email: z.string().trim().email().max(254).toLowerCase(),
  password: z.string().min(8).max(128),
  firstName: z.string().trim().min(1).max(100),
  lastName: z.string().trim().min(1).max(100),
  phone: z.string().trim().max(32).optional(),
}).strict();
export type RegisterBody = z.infer<typeof registerBodySchema>;


  
web/src/features/auth/schemas.ts (client)


const registerBaseSchema = z.object({
  email: emailSchema,
  password: passwordSchema,
  firstName: nameSchema('First name'),
  lastName: nameSchema('Last name'),
  phone: phoneSchema.optional(),
});
export const registerBodySchema = registerBaseSchema;
export type RegisterBody = z.infer<typeof registerBodySchema>;


  

    Different file, different helpers, same shape. The client form schema
    extends the base with confirmPassword; the API body schema
    does not. Each side names what it needs locally without leaking concerns
    across the boundary.
  


  
Why not share an npm package?


  

    You could. Some projects publish a @org/api-contracts
    package and import it in both. Trade-offs:
  


  

    
Parallel schemas (this project)Shared package

    
      
CouplingNone — deploy either side any timeTight — bump shared, redeploy both

      
Drift riskReal — caught at runtime by Zod parseCompile-time, caught immediately

      
Build complexityNoneMonorepo tooling, version bumps

      
Best forSmall/medium teamsMany clients sharing one API

    
  


  

    For a teaching project (and many production projects) the parallel
    approach is simpler, and Zod's runtime parse on the client gives you
    the safety net you'd otherwise rely on compile-time coupling for.
  


  
The four mirror points


  

    The schemas at these four boundaries should always be checked together
    when a feature changes:
  


  
    
    
  1. 
      Request body shape. Server DTO defines what it
      accepts. Client form schema defines what it collects. They differ
      (e.g., confirmPassword) but the API-bound fields must
      match.
    
    2. 
    
  2. 
      Response body shape. Server returns
      PublicUser; client parses with publicUserSchema.
      A new field on the server is invisible until the client schema is
      updated.
    
    3. 
    
  3. 
      Query params shape. Pagination, search, filters.
      Easy to forget because they live in the URL.
    
    4. 
    
  4. 
      Enum values. RoleName, Gender,
      etc. Both sides have their own copy — check the values match.
    
    5. 
  


  

    Why Zod parses both ways?
    Inputs are obvious: refuse junk. But the client also parses
    responses: if the server promised roles: RoleName[]
    and now sends roles: number[], the parser throws and the
    bug surfaces immediately. Without that runtime check, you'd discover
    the drift somewhere in a component, three clicks later, with no clue
    why.
  


  
A small discipline: name the schema after the type



publicUserSchema  → PublicUser
tokenPairSchema   → TokenPair
loginFormSchema   → LoginFormValues


  

    A boring convention, but consistency makes the codebase navigable. If
    you know one name, you know the other.

    Coverage is a means, not a goal. Test the logic — the

    deserves one integration test, not a mock.

    Don't mock the database in integration tests. If your migration drops a
    column the mock can't notice. The repository layer's whole job is to
    talk to a real database; mocking it defeats the test's purpose. Use a
    real (test-scoped) database, even if it's slower.

Security checklist

    
Each box ticked is a class of bug you didn't ship.

  


  

    Web security is not a single feature; it's a collection of small
    disciplines, each defending against one class of mistake. The checklist
    below is what this project does and why. Tick each item against your
    own future projects.
  


  
Transport & identification

  
    
    
  • 
      HTTPS only in production. helmet() sets
      Strict-Transport-Security, which tells browsers to
      refuse plain HTTP for a year.
    
    • 
    
  • 
      No X-Powered-By.
      app.disable('x-powered-by'). Stops shouting your stack at
      every scanner on the internet.
    
    • 
    
  • 
      CORS allowlist. Browser-side cross-origin requests
      only from explicit origins. Never * with credentials.
    
    • 
  


  
Input handling

  
    
    
  • 
      Body limit. express.json({ limit: '100kb' }).
      Anything bigger is denied at the boundary — no DoS via giant
      JSON.
    
    • 
    
  • 
      Zod validation everywhere. Every request body, every
      query string, every path param goes through a schema. The schemas
      use .strict() so unknown fields are rejected, blocking
      mass-assignment.
    
    • 
    
  • 
      Prisma parameterises queries. No string-concatenated
      SQL means no SQL injection in normal code paths. (If you use
      $queryRaw, use $queryRaw`...` with template
      literals, never $queryRawUnsafe.)
    
    • 
  


  
Authentication

  
    
    
  • 
      Bcrypt with cost ≥ 12. Hashing is intentionally slow.
      Per-user salt prevents rainbow-table reuse.
    
    • 
    
  • 
      Timing-equalised login. Missing user vs wrong password
      take the same time. No oracle leaking which emails exist.
    
    • 
    
  • 
      JWTs signed with HS256. The signing secret is at
      least 32 characters, lives only in env, never in code or logs.
    
    • 
    
  • 
      Different secrets for access and refresh. Enforced at
      env-validation time so it cannot drift.
    
    • 
    
  • 
      Refresh tokens stored hashed (SHA-256). DB leak does
      not yield usable tokens.
    
    • 
    
  • 
      Refresh-token rotation with reuse detection. One use
      per token; reuse revokes every active session.
    
    • 
  


  
Authorisation

  
    
    
  • 
      Server enforces all rules. Client hides buttons for
      UX, but every endpoint re-checks roles + ownership.
    
    • 
    
  • 
      Side effects on revocation. Password change and
      account deletion revoke all refresh tokens for that user.
    
    • 
  


  
Rate limiting

  
    
    
  • 
      Global limit (100 req/min/IP) on every endpoint.
    
    • 
    
  • 
      Tighter limits on /auth/login (5 / 15
      min), /auth/register (3 / hour), /auth/refresh
      (30 / min).
    
    • 
  


  
Output handling

  
    
    
  • 
      Same error shape everywhere. Production hides 5xx
      messages and stacks; clients see "Internal server error"
      with a requestId to quote at support.
    
    • 
    
  • 
      React escapes by default. Interpolating a string into
      JSX ({userName}) is XSS-safe. Avoid
      dangerouslySetInnerHTML unless you have a real reason and
      sanitised input.
    
    • 
  


  
Operational

  
    
    
  • 
      Env validated at startup. Bad config fails the
      deployment instead of crashing later.
    
    • 
    
  • 
      Secrets never logged. Logger pulls explicit fields,
      never whole bodies.
    
    • 
    
  • 
      Request IDs everywhere. Bug reports point at exact
      log lines.
    
    • 
    
  • 
      Graceful shutdown. No interrupted writes, no
      poisoned DB connections.
    
    • 
  


  

    Pitfall
    The most common mistake in web security is not a missing
    defence — it is a defence that exists only on the client. A
    role check in the React app that's missing on the server is worse
    than no check at all, because it hides the vulnerability from a
    casual reviewer.
  


  
What this project does not ship (yet)

  
    
    
  • CSRF protection — not needed because the client uses bearer tokens, not cookies. The day refresh tokens move to HttpOnly cookies, this changes.
    • 
    
  • 2FA / WebAuthn — out of scope.
    • 
    
  • Email verification flow — the column exists (emailVerifiedAt) but the flow is left as an exercise.
    • 
    
  • Audit log — the who-changed-what log is deferred.
    • 
  


  

    Note
    Not implemented does not mean not important. It means a future
    iteration. The project is honest about its scope.

The client's URL is not in CORS_ORIGINS on the server.

Appendix A

    
Glossary

    
Terms that appeared in this book, in alphabetical order.

  


  

    
Access token

    
Short-lived JWT carried in the Authorization header. Verified statelessly. 15 min in this project.


    
AppError

    
Project-specific base class for thrown errors. Carries statusCode, code, details, isOperational. Subclassed per HTTP status.


    
Bcrypt

    
Deliberately-slow password hash with a per-user salt. Cost controlled by "rounds" (12 here).


    
Bearer token

    
Any string presented in Authorization: Bearer <token>. Whoever holds it gets access — hence "bearer". Treat like a password.


    
Boundary

    
Where untrusted data enters trusted code (request body, env vars, DB read). Validation lives at boundaries.


    
Controller

    
Thin layer between Express and the service. Reads req, calls service, writes res. Holds no business rules.


    
CORS

    
Cross-Origin Resource Sharing. Browser policy that blocks JS at origin A from calling API at origin B unless B explicitly allows A.


    
DTO

    
Data Transfer Object. The shape of a request or response, defined as a Zod schema + derived TypeScript type in one file.


    
ErrorBoundary

    
React component that catches uncaught render errors in its subtree and shows a fallback UI instead of a blank page.


    
Express

    
Minimal Node.js HTTP framework. Middleware-based. Used by the server in this project.


    
HMR

    
Hot Module Replacement. Vite/dev-server pushes code changes to the browser without a full reload. Component state often survives.


    
Integration test

    
A test that exercises real layers together (HTTP + service + DB). Slower, higher confidence.


    
Invalidation

    
Telling TanStack Query that a cached query is stale and should refetch.


    
JWT

    
JSON Web Token. header.payload.signature. Tamper-evident envelope of claims.


    
Loader (React Router)

    
A function that runs before a route renders. Returns data or redirects.


    
Mass assignment

    
An attack where a client sends extra fields hoping the server will save them blindly (role: "ADMIN"). Prevented by strict input schemas.


    
Middleware

    
An Express function that runs in the request pipeline. Receives (req, res, next); can respond, mutate, or pass through.


    
Migration

    
A versioned SQL script that evolves the database schema. Prisma writes one when you change schema.prisma.


    
N+1 query

    
Anti-pattern: one query for a list, then N more for each item. Solved by include (a JOIN) or batched fetches.


    
Operational error

    
An error we expected and handled (validation, 404, conflict). Opposite: programmer error / bug.


    
ORM

    
Object-Relational Mapper. Generates a typed client from a schema; you write methods instead of SQL.


    
Prisma

    
The ORM used in this project. Schema-first, typed, supports migrations.


    
Refresh token

    
Long-lived secret stored hashed in the DB. Exchanged for a fresh access token. Single-use; rotation on every use.


    
Repository

    
The layer that talks to the database. Only file in a module that imports the Prisma client.


    
Requester context

    
{ id, roles } for the authenticated caller. Passed from controller to service so service can authorise.


    
Request ID

    
A UUID attached to each request and threaded through logs, errors, and response headers. Makes one bug report ≈ one greppable string.


    
Rotation (token)

    
Issuing a new refresh token on every use and revoking the old one. Single-use refresh tokens.


    
Service

    
The brain of a module. Business rules, authorisation, orchestration. Throws AppError subclasses.


    
SPA

    
Single-Page Application. One HTML file, all subsequent navigation done client-side by a router.


    
Soft delete

    
Setting deletedAt instead of removing the row. Reversible; preserves history.


    
StrictMode (React)

    
Development helper that double-invokes effects and renders to expose impure code.


    
TanStack Query

    
Library for managing server state in React. Caches, dedupes, refetches.


    
Timing oracle

    
An attack vector where response time leaks information (e.g., whether an email is registered). Defended against by constant-time logic.


    
TypeScript

    
JavaScript plus a static type checker. Types vanish at runtime.


    
Vite

    
Bundler / dev server used in the client. ESM-native, fast HMR.


    
Vitest

    
Vite-friendly test runner. Used in both halves of this project.


    
Zod

    
Runtime schema library for TypeScript. Validates and parses untrusted data; derives static types.


    
Zustand

    
Minimal store library for React. No provider needed; selectors avoid over-rendering.

  







  

    
Appendix B

    
Lab exercises

    
End-to-end work to consolidate everything in the book.

  


  

    Pick one per week. Each lab spans both halves of the codebase; that is
    the point. The hardest part of full-stack work is not learning a tool;
    it is keeping the two sides in lockstep through a change.
  


  
Lab 1 — Email verification

  

    The User model already has emailVerifiedAt.
    Wire up a flow:
  

  
    
    
  • On register, generate a one-time verification token (similar to refresh tokens: random bytes + SHA-256 hash in DB).
    • 
    
  • Add POST /auth/verify-email taking { token }. Mark the user verified.
    • 
    
  • For now, log the verification link to the console instead of sending email.
    • 
    
  • Client: add a /verify-email route that posts the token from the URL and shows success or failure.
    • 
  


  
Lab 2 — Add the "Subjects" feature

  

    A simple CRUD slice across the whole stack:
  

  
    
    
  • Prisma model Subject with name, code, description?.
    • 
    
  • Module on the server with the four layers (routes/controller/service/repository) and full CRUD.
    • 
    
  • RBAC: admins create/update/delete; everyone authenticated can list.
    • 
    
  • Client feature with list page, create form, edit/delete.
    • 
    
  • Tests: at least one service unit test and one integration test.
    • 
  


  
Lab 3 — Attendance marking

  

    Bring two existing models together:
  

  
    
    
  • POST /api/v1/attendance accepting { studentId, date, status }.
    • 
    
  • Rule: only ADMIN or a TEACHER may mark; a TEACHER may mark only for students in classes they teach. (Yes — the service layer earns its keep here.)
    • 
    
  • The record's recordedBy is the requester. Don't let the client choose.
    • 
    
  • Client: a table of students with a status dropdown and a "Save" button per row.
    • 
  


  
Lab 4 — Audit logging

  

    Capture who-changed-what for the user resource:
  

  
    
    
  • Prisma model AuditEvent with actorId, entityType, entityId, action, diff: Json, createdAt.
    • 
    
  • Service wraps every write with an audit insert in the same transaction.
    • 
    
  • Endpoint GET /api/v1/users/:id/audit — admin-only.
    • 
    
  • Client: an "Audit" tab on user detail page.
    • 
  


  
Lab 5 — "Forgot password"

  

    A complete out-of-band reset flow:
  

  
    
    
  • POST /auth/forgot-password with { email }. Always responds 204 (no email enumeration).
    • 
    
  • If the user exists, generate a one-time token (15-minute TTL, hashed in DB) and log the link.
    • 
    
  • POST /auth/reset-password with { token, newPassword }. On success, revoke all refresh tokens for the user.
    • 
    
  • Client: a request page (form with email) and a reset page (form with password + confirm).
    • 
  


  
Lab 6 — Pagination on every list

  

    The users list already paginates. Audit the other planned lists and:
  

  
    
    
  • Define a consistent query schema: { page, pageSize, sortBy, sortOrder, search?, ... }.
    • 
    
  • Define a consistent response: { items, totalCount, page, pageSize, totalPages }.
    • 
    
  • Client: a shared <Pagination /> + <DataTable /> contract every list uses.
    • 
  


  

    Note
    For each lab, write the tests first (red), make them pass (green),
    then clean up (refactor). The TDD habit is more valuable than any
    specific lab.
